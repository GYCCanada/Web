**Verdict:** Not ship-ready: the engine core is promising, but the assembled Branch 6 PR still misses the whole-PR contract in three load-bearing places: contact/volunteer conditional rendering regresses, registration bypasses CMS-backed `getForm`, and registration parity knowingly diverges from the spec/oracle while tests no longer cover the riskiest render branch.

**BLOCKERS**

1. Contact and volunteer now render inactive conditional fields, which makes normal submissions fail validation.
The plan says the generic renderer should absorb cross-field conditional UI (`/Users/cvr/Developer/personal/gyc/docs/registration-launch-plan.md:221`). The diff deletes the old live `method`-based conditional rendering in contact/volunteer (`/Users/cvr/Developer/personal/gyc/docs/.counsel/form-engine-WHOLE.diff:4395`, `:5808`) and replaces both with `<FormFields>` (`/Users/cvr/Developer/personal/gyc/docs/.counsel/form-engine-WHOLE.diff:4433`, `:5875`). But `FormFields` renders all top-level fields unconditionally (`/Users/cvr/Developer/personal/gyc/app/lib/forms/render.tsx:237`), while optional inactive text fields are invalid if present blank (`/Users/cvr/Developer/personal/gyc/app/lib/forms/decode.ts:223`). I also verified `parseSubmission` preserves blank strings through the form path (`/Users/cvr/Developer/personal/gyc/app/lib/effect/form.ts:143`). Result: `method=email` with blank `phone`, or `method=phone` with blank `email`, fails.

2. Registration is not actually wired to CMS-backed form definitions.
Branch 6 explicitly says definitions should be read through Branch 5’s `getForm` and editing stored `forms/registration.json` should change accepted validation (`/Users/cvr/Developer/personal/gyc/docs/registration-launch-plan.md:217`, `/Users/cvr/Developer/personal/gyc/docs/forms/registration-spec.md:45`). Contact and volunteer do this (`/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/contact.tsx:47`, `/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/volunteer.tsx:44`), but registration imports `defaultRegistrationForm` directly and builds `RegistrationSchema` at module scope (`/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/registration-form.tsx:7`, `:104`). The 2024/2025/2026 route loaders also do not fetch `registration` form content (`/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/2024/form/route.tsx:32`, `/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/2025/form/route.tsx:32`, `/Users/cvr/Developer/personal/gyc/app/routes/($lang)+/2026/form/route.tsx:32`). That violates the “all four registration callers” migration target (`/Users/cvr/Developer/personal/gyc/docs/registration-launch-plan.md:236`).

3. Registration absent-`extra` errors are non-equivalent to the spec/oracle.
The registration spec requires absent attendee `extra` to anchor at `extra.tos` with `registration.form.tos.required` (`/Users/cvr/Developer/personal/gyc/docs/forms/registration-spec.md:77`, `:99`). The default definition comment says the same (`/Users/cvr/Developer/personal/gyc/app/lib/content/pages/defaults.ts:667`). But the generic group presence issue anchors to the first presence-requirable inner field (`/Users/cvr/Developer/personal/gyc/app/lib/forms/decode.ts:327`), and `howDidYouHear` appears before `tos` in the registration definition (`/Users/cvr/Developer/personal/gyc/app/lib/content/pages/defaults.ts:797`, `:862`). The deleted equivalence harness explicitly carved this out as an intentional delta instead of proving identical TranslationKey sets (`/Users/cvr/Developer/personal/gyc/app/lib/forms/equivalence.registration.test.tsx` from commit `98b97c8`: lines 48-59, 493-517). That contradicts the ADR/plan requirement for identical key sets (`/Users/cvr/Developer/personal/gyc/docs/adr/0007-structural-form-builder.md:24`, `/Users/cvr/Developer/personal/gyc/docs/registration-launch-plan.md:230`).

4. Registration render parity does not cover the attendee branch.
The plan calls the registration migration the riskiest commit and requires render-level parity (`/Users/cvr/Developer/personal/gyc/docs/registration-launch-plan.md:285`). The standing test admits SSR only renders the exhibitor branch (`/Users/cvr/Developer/personal/gyc/app/lib/forms/registration-form.test.tsx:78`) and asserts only exhibitor rendered names (`/Users/cvr/Developer/personal/gyc/app/lib/forms/registration-form.test.tsx:119`). Attendee/volunteer fields are checked as default-value keys, not actual rendered controls (`/Users/cvr/Developer/personal/gyc/app/lib/forms/registration-form.test.tsx:152`). That misses the exact class of bug this branch fixed once already, where two rendered names were wrong (`/Users/cvr/Developer/personal/gyc/docs/.counsel/form-engine-WHOLE.diff:4608`).
